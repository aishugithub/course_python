## What this project is

`course_python` is a React + Vite single-page app that delivers an interactive, self-paced Python course for B.Tech students at Sri Ramachandra Faculty of Engineering and Technology. It lives at `C:\aishu\courses\course_python`, is a git repo, and deploys to GitHub Pages at `https://aishugithub.github.io/course_python` via `npm run deploy` (which runs `vite build` then `gh-pages -d dist`). Login and per-student progress tracking go through a Google Apps Script backend, configured in `config/gas.config.js` and called from `src/shell/api.js`.

## Architecture — the four things that matter

The whole app is built around one simple idea: lessons are fully independent, self-contained React components, and one config file decides which ones exist and in what order.

`config/course.config.js` is the only file you edit to change the course's structure. It exports `COURSE_CONFIG`, an object with `courseId`, `courseTitle`, `subtitle`, `batch`, and a `modules` array. Each module has a `moduleId`, `moduleTitle`, and a `units` array of `{ unitId, title }` pairs. `unitId` must exactly match a filename in `src/lessons/` (without the `.jsx` extension) — that's the only link between the config and the actual lesson code.

`src/lessons/UnitX_Y.jsx` — one file per lesson. Every lesson file is completely self-contained: its own local colour-palette constant, its own widgets, zero imports from other lesson files or from each other. This is deliberate — it means any single lesson can be rewritten, deleted, or handed to a different author without touching anything else. `src/shell/App.jsx` discovers every file in this folder automatically using Vite's `import.meta.glob('../lessons/*.jsx')`, so dropping a new `UnitX_Y.jsx` file in and adding a matching entry to `course.config.js` is the entire process for adding a lesson — no other file needs to change.

`src/shell/App.jsx` owns login state, dynamically imports whichever lesson file matches the unit the student clicked, and renders it with two props: `student` (`{ name, rollNo }`) and `onUnitComplete` (a callback the lesson must call exactly once, when the learner finishes, which saves progress via `api.js` and returns them to the Dashboard).

`src/shell/Dashboard.jsx` renders every module and its units as clickable cards, reading straight from `COURSE_CONFIG`. Navigation is free — every unit is always clickable regardless of what's completed — `completedUnits` only controls the ✅/▶️ icon and the overall percentage bar.

## The lesson template every unit follows

Every `UnitX_Y.jsx` file, without exception, has this shape: a header showing `MODULE X › UNIT X.Y` plus a live "N / total done" counter; a thin animated progress bar; a horizontal tab strip of 4-5 sections (the last one is always the quiz); a content card that renders whichever section is active; a "Mark Complete & Continue" button that advances to the next tab (hidden on the quiz tab, since the quiz has its own progression buttons); and a closing multiple-choice quiz with instant colour feedback (green for correct, red for the student's wrong pick) plus a one-line explanation after each answer, ending in a completion screen that shows a score, a contextual message, and previews the next unit or module. Only the quiz's `onComplete` callback calls `onUnitComplete` — that is the single point where a unit's completion is actually persisted.

Every file defines the exact same colour palette (dark, GitHub-inspired): background `#0D1117`, surface `#161B22`, card `#1C2333`, accent blue `#58A6FF`, solid-button blue `#1F6FEB`, success green `#3FB950`, warning yellow `#D29922`, purple `#BC8CFF` for "key insight" callouts, error red `#F85149`, plus orange/teal used for variety in some lessons. Copy this object verbatim into any new lesson file rather than inventing new colours.

## Teaching philosophy — how content gets designed, not just coded

Every widget teaches exactly one idea through interaction — clicking, dragging a slider, stepping through an animation, guessing before revealing — never through a wall of static text. Analogies come before jargon (a program is "a recipe," a variable is "a locker only you can open"). Each unit closes with a quiz that gives immediate feedback and an explanation, not just a score.

The one rule that most shaped Module 4, and should carry forward to every future unit that introduces a new core CS concept: **don't just show the syntax — motivate the need for it first, contrast it with a lower-level language (usually C) to show what Python is actually doing for you, and where relevant, visualise what's happening in memory or at the hardware level.** This came directly from feedback while building Unit 4.2 (Variables & Memory): before naming "variables," the lesson proves why one is needed at all (a football score-counter that literally cannot count without one), then shows an actual row of numbered memory slots being reserved and reused, and only then contrasts `score = 0` in Python against `int score = 0;` in C — framing dynamic typing as a genuine advantage rather than a syntax quirk. Apply this same need → lower-level contrast → memory/hardware visualisation sequence to future concepts: functions and the call stack, lists as contiguous memory, references/aliasing, recursion.

Units are built and delivered one at a time, not batched — each finished `.jsx` file is copied into `src/lessons/` and shown to Aishu immediately, before starting the next unit. Keep doing this for future modules.

**Comment style:** Module 4's files (`Unit4_1.jsx`–`Unit4_4.jsx`) carry heavy, line-by-line explanatory comments — that was requested at the time to document the app's architecture while it was being established. Starting with Module 5, Aishu asked to stop that: the conventions are now well understood and no longer need re-explaining in every file. Do not retroactively strip Module 4's comments; just don't add that density of comments to new files.
